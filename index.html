    RabbitMQ?

    • Message broker 
    • Based on the AMQP open specification 
    • Implemented in Erlang
    • Wicked awesome

    WHAT IS IT GOOD FOR?

    • Decoupled message passing
    • Routing
    • Low-latency communication
    • Configurable guarantees*
      • QoS features
      • Ordered delivery
      • Persistent message queues




    *nothing is perfect

    Basic Concepts

    • Exchange - where messages are sent to
    • Queues - where messages end up
    • Binding - how messages flow from exchange to queue
    • Routing Key - a pattern based rule for routing messages
    • Topic - metadata describing message content
    • Messages - opaque, binary payloads with metadata

    Message Metadata

    • MessageId
    • CorrelationId
    • ContentType
    • Headers
    • Timestamp
    • Type
    • ReplyToAddress
    • Sequence Number, Position, Sequence End

    Message Taxonomies

    Command - tell
    Use sparingly, increases behavioral and temporal coupling

    Query - ask
    Use sparingly, high temporal coupling, latency and failure sensitive

    Event - inform
    Preferred; low temporal, no behavioral coupling, highly resilient to failure


    Reliability Features

    • Publisher confirmation
    • Persistent delivery
    • Consumer acknowledgments
    • Consumer QoS
    • Clustering and mirrored queues
    • Alternative exchanges

    Message Based Decoupling

    Endpoint decoupling - publishers and subscribers are "ignorant" of one another

    Temporal decoupling - publishers can "fire and forget"*


    Behavioral decoupling - the message becomes the contract


    System behaviors can evolve with a few targeted changes vs. complete re-architectures or over-hauls

    Reliability vs. Latency

    • Serializers
    • Serialization formats
    • Persistent delivery
    • Queue mirroring
    • Binding complexity
    • Consumer QoS

    Transformation Cycle

    1 - Create application events

    Application events should have a clear, single meaning






    Transformation Cycle

    2 - Publish events to a top-level exchange

    Always include well-specified topics that provide event context

    Bind catch-all event queue to it for logging / persistence

    Transformation Cycle

    3 - Extract Non-essential Features

    Move as much implementation into individual services activated by messages

    Transformation Cycle

    4 - Create read-models out-of-band

    Introduce document or key/value storage

    Create services which aggregate events to create read-models for the application

    Bonus: consider event sourcing / CQRS

    Transformation Cycle

    5 - Simplify In-Band User Actions

    Reduce application actions to:
    Validation of user data,
    Validation of invariant app rules,
    Read-model access
    Event publishing

    How To Kill A Rabbit


    • Never ack messages in acknowledge mode
    • Avoid consumer QoS limits
    • Too few consumers given publish volume
    • Use consumer reject to implement "selective consumption"
    • Over-bind to a single queue



    Bonus Topic: k-ordered ids

    128 bits
    [ 64 bit timestamp | 48 bit node id | 16 bit sequence id ]
    340 undecillion globally unique ids
    65k ids / ms / node for 584 million years
    Relative total ordering
    0 coordination

    Reference implementations:
    Snowflake - Scala
    Flake - Erlang
    Rustflakes - C#

    Bonus Topic: Isolation

    State mutation must be isolated down to a single execution context (i.e. no parallel mutations)


    This is super hard to do in shared, mutable-state parallelism :(

    Bonus Topic: Vector Clocks

    In highly available systems,
    during network partitions, state can diverge


    Vector clocks provide a causal tracking mechanism to determine which node created which version of state
